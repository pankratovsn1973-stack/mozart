Парадигма выполнения кода в Mozart ERP (ИТОГОВАЯ ВЕРСИЯ 2.0)
Основные принципы
    1. Все расчёты и логика — экземпляры сущностей класса CALC
        ◦ Хранятся в data.entity_instances + data.entity_versions + ext_calc
        ◦ Имеют версионирование, права доступа, историю изменений
        ◦ Могут быть включены/выключены, иметь статус (DEV/PROD)
    2. Способ выполнения определяется по алиасу (префиксу)
        ◦ Префикс указывает, где и как выполняется код
        ◦ Отсутствие префикса — только через менеджер, с явным указанием параметров выполнения
    3. Единый объект выполнения расчётов
        ◦ В L3 (клиент) — app.ORunCalc()
        ◦ В L2 (сервер) — L2_CalcManager
        ◦ В L3 и L2 есть ровно один объект, который умеет вызывать расчёты
    4. Расчёт может выполняться на любом слое
        ◦ На L3 — быстрые локальные расчёты (валидация форм, простые формулы)
        ◦ На L2 — основные бизнес-расчёты (зарплата, налоги, амортизация)
        ◦ На L2 в изолированном ядре — тяжёлые расчёты (отчёты за год)
    5. Единый мост L3 ↔ L2
        ◦ L3 не знает, как отправлять запросы на L2. Это делает один объект L3_DataBridge
        ◦ L2 не знает, откуда пришли запросы. Их принимает один объект L2_DataBridge
        ◦ Все расчёты, выполняемые на L2, идут через этот мост
    6. Безопасность через стандартную систему прав
        ◦ Запуск расчёта = действие EXECUTE над экземпляром CALC
        ◦ Права на расчёт наследуются от роли пользователя через auth.permissions
        ◦ Переданные параметры тоже проверяются (что можно передавать, а что нельзя)

1.1. FormLogic (L3, прямой вызов)

# При открытии формы

Характеристики:
    • Синхронно
    • Блокирует UI (но события короткие)
    • Доступ к L2 только через app.ORunCalc()
    • Выполняется в том же потоке, что и UI

    async=True,

Характеристики:
    • Асинхронно (не блокирует UI)
    • Может выполняться в отдельном ядре/процессе
    • Есть очередь, отмена, прогресс
    • Доступ к БД напрямую

1.4. ActionCalc (L3 или L2, по настройке)
python
# Действие может быть выполнено на L3 (быстро) или на L2 (безопасно)
app.run_action(
    action_alias='act_approve_document',
    level=Level.L2  # или Level.L3, настраивается в конфигураторе
)
Характеристики:
    • Уровень выполнения задаётся в метаданных действия (поле cexecution_level в ext_calc)
    • На L3 — быстро, но рискованно (можно обойти проверки)
    • На L2 — медленнее, но безопасно (все проверки на сервере)

5. Логика загрузки кода в L2/L3
python
def get_calc_code(calc_alias):
    """Возвращает код CALC-сущности"""
    calc = load_calc_instance(calc_alias)
    
    if calc.ccode and calc.ccode.strip():
        # Код в БД — используем его
        return calc.ccode
    elif calc.cfilepath and os.path.exists(calc.cfilepath):
        # Код в файле — читаем
        with open(calc.cfilepath, 'r', encoding='utf-8') as f:
            return f.read()
    else:
        raise Exception(f"Нет кода для расчёта {calc_alias}")
6. Для форм — автоматическое создание CALC
При создании новой формы в конфигураторе:
    1. Создаётся запись в meta.forms
    2. Автоматически создаётся CALC-сущность:
        ◦ calias = form_<form_alias>_logic
        ◦ cname = Логика формы <form_alias>
        ◦ ccode = шаблон кода (пустая заглушка с комментариями)
        ◦ cfilepath = NULL
    3. Всё редактирование кода — через встроенный редактор, который пишет прямо в ccode.
7. Для расчётов — выбор при создании
В конфигураторе при создании CALC-сущности разработчик выбирает:
text
○ Хранить код в базе
○ Хранить код в файле: [________________] (кнопка "Обзор")
Если выбран файл — ccode остаётся пустым, используется cfilepath.
8. Жизненный цикл расчёта
8.1. Регистрация расчёта (в конфигураторе)
    • Создаётся сущность класса CALC (экземпляр + версия)
    • Заполняются поля ext_calc
8.2. Вызов расчёта (из L3)
python
result = app.ORunCalc(
    calc_alias="payroll_calculation",   # алиас расчёта
    level=Level.L2,                      # где выполнять: L2 / L3
    async_mode=False,                    # синхронно / асинхронно
    kernel=Kernel.ISOLATE,               # как выполнять: main / thread / isolate
    params={                             # входные параметры (значения)
        "period": "2025-01",
        "department_id": 123
    },
    ref_params={                         # ссылочные параметры (указатели)
        "document": "invoice:456"
    },
    output_params=["result", "errors"],
    callback=on_calc_finished
)
8.3. Маршрутизация запроса
    • app.ORunCalc проверяет level
    • Если level = L3 → передаёт в L3_CalcManager (локальное выполнение)
    • Если level = L2 → передаёт в L3_DataBridge
L3_DataBridge формирует пакет:
json
{
    "command": "run_calculation",
    "calc_alias": "payroll_calculation",
    "async": false,
    "params": {...},
    "ref_params": {...},
    "output_params": ["result", "errors"],
    "session_id": "uuid"
}
8.4. Приём и выполнение на L2
    • L2_DataBridge получает пакет, проверяет авторизацию
    • Передаёт в L2_CalcManager
    • L2_CalcManager:
        ◦ Находит экземпляр CALC по алиасу
        ◦ Проверяет права пользователя (EXECUTE)
        ◦ Проверяет, активен ли расчёт (lisactive = TRUE)
        ◦ Проверяет параметры по схеме (xparams_json)
        ◦ Загружает код (из ccode или cfilepath)
        ◦ Запускает выполнение
        ◦ Собирает результат
8.5. Возврат результата
    • Результат идёт обратным путём: L2_CalcManager → L2_DataBridge → L3_DataBridge → app.ORunCalc (или callback)
8.6. Асинхронный режим
    • При async_mode=True L3 не ждёт результат
    • Результат приходит через callback
    • Можно отслеживать статус выполнения

10. Безопасность передачи параметров
Проблема: Передаём ссылку на документ "invoice:456" → что, если пользователь передаст документ, на который у него нет прав?
Решение:
    • L2 при получении ref_params проверяет каждый указатель
    • Проверяет права пользователя на READ этого документа
    • Если прав нет — расчёт не запускается, ошибка

13. Итог
Овцы целы, волки сыты.
    • Формы — код в БД, прямой вызов на L3.
    • Расчёты — код в файлах (опционально в БД), через CALC-менеджер на L2.
    • Действия — код в БД, уровень выполнения настраивается.
    • Расчёты без префикса — работают через менеджер с явными параметрами (обратная совместимость).
Всё через одну CALC-сущность.
